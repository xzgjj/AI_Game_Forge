# Figma UI 设计协作规范

最后更新：2026-04-02

## 目标
把当前桌面端工作流整理成一份可在 Figma 中持续协作的 UI 设计基线，确保设计稿、代码和文档描述同一条用户流。

## 逐页核对入口
详细的页面级视觉与交互对照见 [doc/figma_page_review.md](figma_page_review.md)。该文档用于逐页验收，不替代本规范。

## 当前用户流
1. 登录
2. 仪表盘
3. 配置向导
4. AI 画布
5. 返回仪表盘

## 登录页
### 目的
让用户用最少输入完成进入工作台。

### 必须展示
- 邮箱登录表单
- 微信扫码登录
- 手机验证登录
- 邮箱注册
- GitHub / Google 第三方登录
- 错误反馈与加载态

### 验证点
- 任一登录方式都要有清晰的下一步反馈。
- 失败态必须能回到同一登录页继续尝试。

## 仪表盘
### 目的
作为进入项目与工作流的主入口。

### 必须展示
- 顶部主操作区
- 总览
- 项目管理
- 用户设置
- API 统计
- 进入配置向导的入口
- 进入 AI 画布的入口

### 验证点
- 项目管理中的“打开”应进入配置向导。
- 仪表盘中的主按钮应能进入向导和画布。

## 配置向导
### 目的
承载项目设计、Unity 初始化、AI 生成与发布准备的结构化输入。

### 必须展示
- 设计
- 架构
- Unity 初始化
- 核心脚本
- 角色
- 资源
- 迭代
- 发布

### 验证点
- 每一步都要有上一步 / 下一步 / 完成的反馈。
- Unity 初始化与校验状态必须可见。
- 左侧表单和右侧结构化预览要同时保留。

## AI 画布
### 目的
给用户一个可回到仪表盘的轻量生成和历史查看界面。

### 必须展示
- 输入区
- 预览区
- 版本历史
- 返回仪表盘入口

### 验证点
- 点击生成后应有内容变化。
- 历史记录要保留最近生成结果。

## 设计约束
- 使用深色、渐变、分层卡片的视觉语言。
- 不要把 8 个内部阶段画成 8 个用户页面。
- 所有页面必须明确失败态、加载态和返回路径。
- 如果设计与代码不一致，以当前代码中的实际流转为校验基线，再决定是否改代码或改设计。

## 协作方式
1. 先按本文件做 Figma 初稿。
2. 再按 [doc/figma_page_review.md](figma_page_review.md) 逐页核对细节。
3. 每次改动都同步更新 `notes.txt` 和 `diff.md`。
4. 如果页面流变化，先更新本文件，再改代码。
