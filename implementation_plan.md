# implementation_plan.md

最后更新：2026-03-14

## 目标
把“Unity + AI 生成游戏”的 8 步流程落地为可用产品，确保每一步有清晰产物、可回滚、可审计。



## 阶段计划（8 步全量拆解）

说明：每一步为独立页面，必须有明确产物与校验；产物写入 `WizardState` 并可入库。

### Step1：游戏设计
目的：明确游戏方向与约束，形成后续所有生成的输入基准。
理解：这是“需求冻结点”，所有后续产物必须可追溯到这里。

- Step1 UI：游戏类型、核心玩法、目标平台、叙事风格、受众画像。
- 结构化产物：`game_design_manifest.json`（写入 WizardState）。
- 约束字段：平台限制、性能目标、美术与叙事风格锁定。
- 参考素材：支持链接或本地参考图记录（不做资源导入）。

验收：可导出结构化设计产物，并能在 Step2 复用。

### Step2：AI 设计架构
目的：把游戏系统拆解为可生成的模块与脚本结构。
理解：这是“系统蓝图”，决定 AI 生成脚本与资源的边界。

- Step2 UI：系统模块拆分、脚本结构草图、AI 生成范围确认。
- 结构化产物：`architecture_manifest.json`（模块、接口、依赖）。
- 关键约束：玩家控制、敌人 AI、碰撞、状态机必须覆盖。
- 生成策略：为 Step4 提供 prompt 模板与输出结构。

验收：模块清单完整，能驱动 Step4 脚本生成。

### Step3：Unity 项目初始化
目的：建立 Unity 工程骨架并与 App 联动。
理解：这是“载体准备”，后续产物必须能稳定导入该工程。

- Unity Bridge：项目模板、Scene、目录结构生成。
- 写入 `ProjectSettings` 与 `Packages/manifest.json`。
- 结构化产物：`unity_project_manifest.json`（路径、版本、模板）。
- 校验：基础目录、关键文件、Unity 版本一致性检查。

验收：Unity 项目可打开，基础场景可运行。

### Step4：AI 生成核心脚本
目的：生成最小可运行的玩法脚本闭环。
理解：这是“可玩性起点”，必须可编译与可运行。

- 生成内容：玩家控制、敌人 AI、碰撞检测、状态机。
- 产物结构：`Scripts/AIForgeCore/*` + 脚本清单。
- 校验：静态检查 + 可选 BatchMode 编译验证。
- 回写：错误日志回传到 Step4 UI。

验收：脚本可编译，Unity 场景可运行最小玩法。

### Step5：角色生成管线落地
目的：把角色从“文本描述”变成“可导入 Unity 的结构化产物”。
理解：这是角色资产的标准化入口，必须能复用、可追溯、可批量导入。

- 扩展 `WizardState` 角色结构体（基础信息、画像、台词模板、行为标签）。
- Step5 UI：角色列表 + 详情编辑 + 结构化预览。
- 画像导入：本地文件选择、格式/尺寸校验、自动拷贝到资源缓存。
- 画像生成：AI 生成入口 + 参数记录 + 结果绑定。
- 台词模板：占位符系统（如 `{name}` `{mood}` `{scene}`）。
- 角色产物：`character_manifest.json` 或 ScriptableObject JSON。
- Unity 导入：映射 `Assets/AIForge/Characters/` 并生成占位 Prefab。
- 校验：缺失字段、重复命名、非法图片格式阻断进入 Step6。

验收：可创建角色并导入/生成画像，Unity 内可见角色占位资源。

### Step6：资源生成/导入管线
目的：将贴图、音效、Prefab 统一纳入标准目录与清单。
理解：这是资产层的“入口规范 + 导入机制”，决定后续迭代是否可控。

- 目录规范：`Assets/AIForge/Textures/Audio/Prefabs`。
- Step6 UI：资源清单 + 导入入口 + 校验面板。
- 贴图导入：尺寸/格式/命名规则校验，自动生成缩略图。
- 音效导入：格式/时长/采样率校验，统一归档命名。
- Prefab 导入：路径校验、依赖检查、缺失引用提示。
- 资源清单：`resource_manifest.json` 关联 WizardState。
- Unity 注入：写入目录并刷新 AssetDatabase。
- 回滚点：导入失败时回滚到上一次成功版本。

验收：资源导入成功且清单与 Unity 目录一致。

### Step7：玩法迭代闭环
目的：让玩法调整形成可追溯、可回滚、可比较的迭代链。
理解：把“调参”从一次性操作变成可管理版本。

- Step7 UI：参数面板 + 变更记录 + 回滚入口。
- 可调参数集：数值、技能冷却、移动速度、伤害倍率等。
- 变更记录：版本号、操作者、时间、差异摘要。
- 迭代对比：展示前后差异，支持导出改动摘要。
- 回滚机制：按版本恢复历史参数集。
- 编译/运行校验：可选触发 BatchMode 校验并回传错误。

验收：完成一次参数迭代与回滚，历史可复现。

### Step8：打包发布
目的：形成可分发的正式版本并保留发布记录。
理解：这是“可交付性闭环”，必须可回溯与可验证。

- 版本策略：语义化版本 + 构建号 + 发布记录。
- 打包配置：平台选项、图标、资源清单、签名策略。
- 构建产物归档：安装包与变更记录绑定。
- 发布审计：发布人、时间、产物哈希、变更摘要。

验收：至少成功打包一个版本并可回溯构建记录。



## 风险控制
- 生成脚本编译失败必须在 Step4 阶段拦截。
- 资源命名与 GUID 映射必须稳定。
- 文档与实现必须同步，禁止“报告先行、实现缺失”。
