# implementation_plan.md

最后更新：2026-03-24

## 目标
把产品从“线性 8 步向导”重构为“5 个用户页面 + 8 个内部生成阶段”的可落地工作台。

用户负责最少输入、查看结果、编辑节点和版本决策；系统负责工程初始化、内容生成、验证、版本快照与发布记录。

## 产品结构

### 用户可见的 5 个页面

#### 页面1：新建项目
目的：用最少输入创建项目，并冻结需求输入。

用户行为：
- 填写项目名、题材、平台、核心玩法、风格方向。
- 选择一个参考目标或默认模板。
- 点击创建项目。

系统行为：
- 自动创建项目骨架、默认目录和基础配置。
- 生成设计摘要和初始工作区状态。
- 初始化后续页面所需的项目上下文。

用户反馈：
- 显示创建进度、创建结果和项目摘要。
- 成功后直接进入资源页。
- 失败时返回清晰错误和可重试入口。

状态变化：
- `draft` -> `project_created` -> `workspace_ready`

验收：
- 用户无需手动处理 Unity 工程细节即可创建项目。
- 创建结果可恢复，可继续进入后续页面。

#### 页面2：资源生成 / 导入
目的：统一处理资源获取、校验、同步与追踪。

用户行为：
- 导入参考图、音频、Prefab、文本。
- 触发 AI 生成资源或批量归档。
- 查看资源状态与引用关系。

系统行为：
- 自动分类、命名、校验和生成缩略图。
- 写入资源清单和来源记录。
- 同步到 Unity 目录或资源缓存。

用户反馈：
- 显示导入成功、失败原因、重复项和缺失项。
- 展示缩略图、来源、Unity 路径和引用状态。

状态变化：
- `untracked` -> `validated` -> `synced` -> `versioned`

验收：
- 资源进入项目后必须可追踪、可回滚、可重新导入。
- 资源页必须能看到“哪些资源已经生效，哪些还没进入 Unity”。

#### 页面3：内容画布
目的：把背景、角色、台词、脚本节点组织成可编辑图谱。

用户行为：
- 一键新建角色或 NPC。
- 在画布上拖拽、连接和编辑节点。
- 对局部内容重新生成或手动修订。
- 从右上角设置触发最小场景验证。

系统行为：
- 将脚本背景拆成角色卡、台词卡、关系卡和场景卡。
- 根据设定自动分配角色定位、台词模板和背景上下文。
- 生成最小可运行场景的验证入口。

用户反馈：
- 显示节点新增、关系变化、局部重生成结果。
- 显示最小场景验证状态、日志和错误定位。
- 让用户直观看到“生成后长什么样”。

状态变化：
- `blank_canvas` -> `content_graph_ready` -> `node_generated` -> `scene_validated`

验收：
- 可从空项目生成第一批可编辑内容节点。
- 角色与台词不再只是表单字段，而是可操作对象。

#### 页面4：协作优化
目的：围绕已有内容做版本化迭代和修正。

用户行为：
- 修改角色、台词、数值、脚本或关系。
- 对比修改前后差异。
- 按验证结果决定接受、继续迭代或回滚。

系统行为：
- 创建修订稿，不覆盖旧版本。
- 计算 diff 和影响范围。
- 触发局部验证，必要时提示冲突或风险。

用户反馈：
- 显示前后对比、差异摘要和受影响节点。
- 显示验证通过与否，以及回滚入口。

状态变化：
- `current_version` -> `revision_created` -> `diff_reviewed` -> `validated_or_rolled_back`

验收：
- 任意一次调整都带版本号、差异和验证结果。
- 用户可以明确知道“改了什么、影响了什么、能不能回退”。

#### 页面5：发布 / 版本记录
目的：查看版本、构建、发布与回滚。

用户行为：
- 选择候选版本。
- 填写发布说明。
- 触发构建或导出。
- 回滚到历史版本。

系统行为：
- 生成 release snapshot。
- 记录构建结果、哈希和审计信息。
- 保留可回溯的发布链路。

用户反馈：
- 显示时间线、构建状态、产物摘要和回滚点。
- 显示发布成功、失败原因或待修复项。

状态变化：
- `candidate` -> `building` -> `released` -> `archived`

验收：
- 至少可以成功保留一个可回溯的发布版本。
- 用户可按版本回到历史快照。

### 内部 8 阶段流水线

说明：以下阶段是实现和校验的内部流水线，不等同于用户页面数量。

1. 项目启动与工程骨架
2. 需求冻结与设计摘要
3. 系统拆分与内容图谱初始化
4. 资源目录与清单建立
5. 角色 / 台词 / 脚本生成
6. Unity 最小场景验证
7. 版本快照与协作迭代
8. 发布打包与审计归档

## 核心对象模型

### 必须具备的状态层次
- `ProjectState`：项目基础信息、目标、平台、风格和当前生命周期状态。
- `WorkspaceState`：当前工作区打开的页面、选中的节点、临时草稿和验证状态。
- `ContentNode`：角色、台词、场景、脚本片段、资源引用等可编辑对象。
- `ContentEdge`：节点之间的关系，例如“角色属于场景”“台词绑定角色”“资源引用节点”。
- `AssetRecord`：资源来源、类型、路径、校验结果、Unity 映射和版本信息。
- `VersionSnapshot`：某一时刻的完整快照，支持 diff 和回滚。
- `ReleaseRecord`：发布版本、构建信息、哈希、审计记录和回退目标。
- `ValidationReport`：页面级、局部级和发布级校验结果。

### 数据持久化要求
- 所有结构化数据都要写入 JSON 和数据库。
- 资源必须带 manifest 和 GUID 追踪。
- 内容节点必须保留原始输入、AI 输出和人工修订记录。
- 版本快照不能被覆盖，只能追加新版本或回滚到旧版本。

## 页面扩张规则

### 新增内容的处理方式
1. 先创建草稿，不直接覆盖正式版本。
2. AI 生成或人工编辑后生成 diff。
3. 校验通过后再进入当前工作集。
4. 每次提交都形成 snapshot。

### 版本迭代的扩张方式
1. 页面功能优先扩展对象能力，不优先堆新的表单字段。
2. 页面1 扩展的是项目约束和设计摘要。
3. 页面2 扩展的是资源健康、依赖图和批量导入。
4. 页面3 扩展的是画布节点、关系、局部重生成和预览。
5. 页面4 扩展的是 diff、验证、回滚和分支修订。
6. 页面5 扩展的是版本树、发布记录和构建审计。

## 可落地性 Review

### 当前实现的不足
- 现有前端仍然偏向线性向导，需要改成工作台式页面结构。
- 当前 `WizardState` 还是扁平快照，不能直接支撑画布节点和版本分支。
- Unity Bridge 现在主要覆盖初始化、UPM 注入和校验，还缺少资源同步、最小场景验证与回滚流程。
- 版本管理还没有真正做到“内容对象级 diff + snapshot + release”。

### 结论
- 这个系统可以完整实现，但前提是先重构状态模型，再重构页面交互。
- 不能继续沿用“8 个表单页”的理解来做后续开发。

## 落地节奏

说明：当前先更新方案，不急着进入代码改造。实现必须按依赖顺序推进，避免先做 UI 再返工状态模型。

### 阶段A：方案冻结
目标：统一页面职责、对象模型和版本语义。

- 冻结 5 个页面的职责边界。
- 冻结 Project / Workspace / ContentNode / VersionSnapshot / ReleaseRecord 的数据定义。
- 冻结资源、版本、审计的命名规则。

验收：
- 所有人对“页面做什么、内部阶段做什么”理解一致。
- 后续代码改造有明确的数据边界。

### 阶段B：状态底座
目标：先把能承载页面的数据模型和持久化打通。

- 将 `WizardState` 拆分为项目级、工作区级、内容级和版本级对象。
- 让 SQLite 支持项目、节点、资源、快照、发布记录的分表或分层存储。
- 为 diff、回滚和审计预留字段。

验收：
- 一个项目可以保存多份快照，而不是只覆盖最后一次状态。
- 内容节点和版本记录可以独立查询。

### 阶段C：工作台壳
目标：把单一 wizard 改成 5 页工作台导航。

- 页面1-5 只负责导航、状态展示和基础动作入口。
- 先把页面切换、返回、恢复、草稿保存做好。
- 暂时不追求所有页面都接入完整生成逻辑。

验收：
- 用户可以在 5 页之间切换，并保留工作区上下文。
- 页面之间的数据不会丢失。

### 阶段D：内容与资源闭环
目标：打通页面2 和页面3 的真实对象流转。

- 页面2 先做资源导入、校验和 Unity 映射。
- 页面3 再做角色卡、台词卡、关系线和最小场景验证。
- 所有新增内容都必须生成 diff 和草稿，不直接覆盖。

验收：
- 资源能进项目，内容能变成节点，节点能形成可视化关系。
- 能看到“生成结果”和“生效状态”的区别。

### 阶段E：版本与发布
目标：让协作优化和发布记录真正可回滚、可审计。

- 页面4 做版本 diff、局部重生成、回滚。
- 页面5 做发布记录、构建快照、发布审计。
- Unity 导入失败、验证失败和发布失败必须可追踪到具体版本。

验收：
- 至少能走通一次“修改 -> diff -> 验证 -> snapshot -> 发布 -> 回滚”闭环。

## 近期实现优先级
1. 先冻结方案与对象模型，不急着改 UI 细节。
2. 再拆状态底座，确保能存项目、节点、资源、快照和发布记录。
3. 然后重构前端工作台壳，保证 5 页导航和上下文保留。
4. 接着补页面2 / 页面3 的资源与内容闭环。
5. 再补页面4 / 页面5 的版本与发布闭环。
6. 最后才做 Unity 更深层的资源同步、回滚和批量验证优化。

## 风险控制
- 生成内容不能直接覆盖历史版本。
- 新页面必须能说明状态变化、用户反馈和回滚路径。
- Unity 导入失败必须可定位、可重试、可回滚。
- 文档与实现必须同步更新，禁止继续依赖旧的 8 步表单定义。
